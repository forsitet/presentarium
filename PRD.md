## 1. Обзор продукта и цели

### 1.1 Описание

ВП ПИО — веб-приложение для проведения интерактивных опросов, квизов и брейнштормов в режиме реального времени. Аналог Kahoot/Mentimeter с возможностью гибкой настройки сценариев, собственным хостингом и независимостью от внешних сервисов.

### 1.2 Проблема

Существующие решения (Kahoot, Mentimeter, Slido) обладают:

- Ограниченной функциональностью в бесплатных тарифах
- Высокой стоимостью лицензирования для корпоративного использования
- Сложностью интеграции с внутренними системами
- Зависимостью от внешних серверов и политик использования
- Отсутствием контроля над данными

### 1.3 Цели

- Предоставить организаторам инструмент для создания и проведения интерактивных опросов с гибкой настройкой
- Обеспечить участникам мгновенное подключение без регистрации
- Поддержать режимы квиза, облака слов и брейншторма
- Обеспечить сбор статистики и экспорт результатов
- Гарантировать работу в реальном времени для сотен одновременных участников

### 1.4 Метрики успеха

- Поддержка до 500 одновременных участников в одной комнате без деградации
- Задержка доставки вопроса участникам < 500 мс
- Автоматическое переподключение WebSocket при обрыве < 3 сек
- Время загрузки страницы участника < 2 сек

## 2. Целевая аудитория

### 2.1 Организатор

- Преподаватели, тренеры, HR-специалисты, ведущие мероприятий
- Используют десктопный браузер для создания опросов и управления сессией
- Нуждаются в простом интерфейсе для создания вопросов и мониторинге результатов в реальном времени

### 2.2 Участник

- Студенты, сотрудники компаний, посетители мероприятий
- Подключаются преимущественно с мобильных устройств по QR-коду или ссылке
- Не требуют регистрации — только ввод имени/никнейма
- Ожидают мгновенный отклик и минимальное количество шагов для участия

## 3. Основные функции и функциональность

### 3.1 Аутентификация и управление аккаунтом организатора

**Описание:** Регистрация и вход по email/пароль. Участники входят анонимно — только имя при подключении к комнате.

**Детали реализации:**

- Регистрация: email (string, unique, max 255), password (string, min 8 символов, bcrypt hash), name (string, max 100)
- Аутентификация: JWT-токены (access token — 15 мин, refresh token — 7 дней)
- Access token передаётся в заголовке `Authorization: Bearer <token>`
- Refresh token хранится в httpOnly cookie
- Эндпоинты: `POST /api/auth/register`, `POST /api/auth/login`, `POST /api/auth/refresh`, `POST /api/auth/logout`
- Восстановление пароля: отправка ссылки на email с одноразовым токеном (TTL 1 час)

**Acceptance criteria:**

- Организатор может зарегистрироваться по email, войти и выйти из системы
- При вводе невалидного email или короткого пароля отображается ошибка валидации
- После истечения access token автоматически обновляется через refresh token
- Невалидный refresh token приводит к разлогиниванию
- Пароль хранится только в виде bcrypt hash

**Технические соображения:**

- Библиотека: `golang-jwt/jwt/v5` для JWT
- Библиотека: `golang.org/x/crypto/bcrypt` для хэширования паролей
- Rate limiting на эндпоинтах аутентификации: не более 5 попыток в минуту на IP

---

### 3.2 Управление опросами (CRUD)

**Описание:** Организатор создаёт, редактирует, копирует и удаляет опросы. Каждый опрос содержит набор вопросов с настраиваемыми параметрами.

**Детали реализации:**

- Поля опроса: title (string, max 200), description (string, max 1000), scoring_rules (enum: `none`, `correct_answer`, `speed_bonus`), question_order (enum: `sequential`, `random`), created_at, updated_at
- Каждый опрос принадлежит одному организатору (owner_id → users.id)
- Копирование создаёт полный дубликат опроса со всеми вопросами
- REST API: `GET/POST /api/polls`, `GET/PUT/DELETE /api/polls/:id`, `POST /api/polls/:id/copy`

**Acceptance criteria:**

- Организатор может создать опрос с названием, описанием и настройками
- Организатор может отредактировать любой свой опрос
- Копирование создаёт независимую копию с пометкой «(Копия)» в названии
- Организатор не может видеть или редактировать чужие опросы
- Удаление опроса каскадно удаляет все его вопросы

---

### 3.3 Банк вопросов

**Описание:** Формирование набора вопросов внутри опроса. Поддерживаемые типы: одиночный выбор, множественный выбор, открытый текстовый ответ, вопрос с изображением, облако слов, брейншторм.

**Типы вопросов:**

| Тип                                | Код              | Описание                                                           | Правильный ответ |
| ------------------------------------- | ------------------- | -------------------------------------------------------------------------- | ------------------------------- |
| Одиночный выбор         | `single_choice`   | 2–6 вариантов, один правильный                     | Да                            |
| Множественный выбор | `multiple_choice` | 2–6 вариантов, несколько правильных           | Да                            |
| Открытый ответ           | `open_text`       | Свободный текстовый ввод (max 500 символов)  | Нет                          |
| С изображением           | `image_choice`    | Как single/multiple, но с изображением в вопросе | Да                            |
| Облако слов                 | `word_cloud`      | Короткий текстовый ответ (max 50 символов)   | Нет                          |
| Брейншторм                  | `brainstorm`      | Свободный ввод идей + голосование              | Нет                          |

**Поля вопроса:**

- poll_id (FK → polls.id)
- type (enum, см. таблицу)
- text (string, max 500)
- image_path (string, nullable) — путь к изображению на сервере
- time_limit_seconds (int, 5–300, default 30)
- points (int, default 100)
- position (int) — порядковый номер вопроса
- options (JSON array для choice-типов): `[{text: string, is_correct: boolean}]`

**Загрузка изображений:**

- Эндпоинт: `POST /api/upload/image`
- Форматы: JPEG, PNG, WebP
- Максимальный размер: 5 МБ
- Хранение: локальная файловая система в директории `./uploads/images/`
- Имя файла: `{uuid}.{ext}` для предотвращения конфликтов

**Acceptance criteria:**

- Организатор может добавить вопрос каждого типа с корректными параметрами
- Изображения загружаются и отображаются в вопросе
- Порядок вопросов можно изменять drag-and-drop
- Валидация: choice-вопросы требуют минимум 2 варианта, минимум 1 правильный ответ
- Время на вопрос настраивается в диапазоне 5–300 секунд

---

### 3.4 Запуск сессии (комнаты)

**Описание:** Организатор запускает сессию опроса, создавая «комнату». Генерируются уникальный код, ссылка и QR-код для подключения участников.

**Детали реализации:**

- Код комнаты: 6-значный числовой код (проверка уникальности среди активных комнат)
- Ссылка: `https://{domain}/join/{room_code}`
- QR-код: генерируется на фронтенде из ссылки (библиотека `qrcode.react`)
- Состояния комнаты: `waiting` → `active` → `showing_question` → `showing_results` → `finished`
- Эндпоинты: `POST /api/rooms` (создать), `PATCH /api/rooms/:code/state` (сменить состояние)
- Одновременно у организатора может быть только одна активная комната на опрос

**Acceptance criteria:**

- При запуске генерируется уникальный 6-значный код
- Отображаются код, ссылка и QR-код на экране ожидания
- Организатор видит список подключённых участников в реальном времени
- Организатор может начать опрос, переключать вопросы и завершить сессию
- При завершении комната переходит в статус `finished` и код освобождается

---

### 3.5 Подключение участников

**Описание:** Участник подключается к комнате по коду, QR-коду или ссылке. Вводит только имя. Подключение происходит через WebSocket.

**Детали реализации:**

- Страница подключения: `GET /join` (ввод кода) или `GET /join/{room_code}` (прямой вход)
- Участник вводит имя (string, 2–30 символов) и подключается
- WebSocket endpoint: `ws://{domain}/ws/room/{room_code}?name={name}`
- При подключении участнику назначается session_id (UUID), хранится в памяти сервера
- Статусы участника: `connected`, `disconnected`
- При обрыве соединения: автоматическое переподключение клиента (exponential backoff, max 5 попыток)
- При переподключении: восстановление по session_id (передаётся в query параметре)

**Acceptance criteria:**

- Участник может войти по коду, QR или ссылке, введя только имя
- При подключении организатор видит нового участника в списке
- При обрыве WebSocket клиент автоматически переподключается
- После переподключения участник видит актуальное состояние комнаты
- Дублирование имён допускается (различение по session_id)

### 3.6 Проведение опроса в реальном времени

**Описание:** Организатор последовательно показывает вопросы. Участники видят вопрос, таймер и отправляют ответ. Ответы принимаются только в пределах таймера.

**WebSocket-протокол (сообщения):**

Сервер → Клиенты:

```json
{"type": "question_start", "data": {"question_id": 1, "text": "...", "type": "single_choice", "options": [...], "time_limit": 30, "image_url": null}}
{"type": "question_end", "data": {"question_id": 1}}
{"type": "timer_tick", "data": {"remaining": 25}}
{"type": "answer_accepted", "data": {"question_id": 1}}
{"type": "results", "data": {"question_id": 1, "distribution": {...}, "correct_option": 0}}
{"type": "leaderboard", "data": {"rankings": [{"name": "...", "score": 100, "position": 1}]}}
{"type": "session_end", "data": {"final_leaderboard": [...]}}
```

Клиент → Сервер:

```json
{"type": "submit_answer", "data": {"question_id": 1, "answer": 0}}
{"type": "submit_text", "data": {"question_id": 1, "text": "ответ"}}
{"type": "submit_vote", "data": {"question_id": 1, "idea_id": "uuid"}}
```

**Валидация ответов:**

- Ответ принимается только если вопрос активен (таймер не истёк)
- Один ответ на вопрос от участника (повторная отправка отклоняется)
- Проверка формата: для choice — индекс варианта, для text — строка ≤ 500 символов
- Timestamp ответа фиксируется на сервере (не доверяем клиенту)

**Acceptance criteria:**

- Вопрос отображается одновременно у всех участников
- Таймер синхронизирован через серверные тики
- После истечения таймера ответы не принимаются
- Участник получает подтверждение приёма ответа
- Повторная отправка ответа на тот же вопрос отклоняется
- Организатор видит количество ответивших в реальном времени

### 3.7 Система подсчёта баллов и лидерборд

**Описание:** Подсчёт баллов по настроенным правилам. Формирование рейтинговой таблицы.

**Правила начисления баллов:**

| Режим         | Формула                              | Описание                                                           |
| ------------------ | ------------------------------------------- | -------------------------------------------------------------------------- |
| `none`           | 0                                           | Баллы не начисляются (режим опроса)           |
| `correct_answer` | `points`                                  | Фиксированные баллы за правильный ответ |
| `speed_bonus`    | `points × (remaining_time / total_time)` | Больше баллов за быстрый ответ                   |

**Детали реализации:**

- Баллы рассчитываются на сервере сразу при получении ответа
- Лидерборд пересчитывается после каждого вопроса
- Лидерборд отправляется всем клиентам после завершения каждого вопроса
- Для `speed_bonus`: `score = base_points × (time_remaining / time_limit)`, минимум 10% от base_points

**Acceptance criteria:**

- Баллы начисляются корректно по выбранному режиму
- Лидерборд отображается после каждого вопроса (топ-5 + позиция текущего участника)
- Итоговый лидерборд показывается при завершении сессии
- При одинаковых баллах выше тот, кто ответил быстрее

---

### 3.8 Облако слов

**Описание:** Участники отправляют короткие текстовые ответы. Ответы агрегируются и отображаются в виде облака слов с визуализацией частотности.

**Алгоритм обработки:**

1. Получить текстовый ответ участника (max 50 символов)
2. Нормализация: trim → toLowerCase → удаление знаков препинания
3. Применить фильтр нецензурной лексики (словарь)
4. Подсчёт частот нормализованных ответов
5. Отправить обновлённые данные частот организатору
6. Фронтенд рендерит облако слов (библиотека `react-wordcloud`)

**Acceptance criteria:**

- Ответы «Москва», «москва», «МОСКВА» считаются одним словом
- Облако обновляется в реальном времени по мере поступления ответов
- Размер слова пропорционален частоте
- Нецензурные слова фильтруются и не попадают в облако
- Организатор может вручную скрыть/удалить конкретные ответы

---

### 3.9 Режим «Брейншторм»

**Описание:** Участники вводят идеи в свободной форме. После сбора идей проводится голосование, результаты ранжируются.

**Фазы:**

1. **Сбор идей** — участники отправляют текстовые идеи (max 300 символов, до 5 идей на участника)
2. **Голосование** — каждый участник может проголосовать за идеи других (до 3 голосов)
3. **Результаты** — ранжирование идей по количеству голосов

**Acceptance criteria:**

- Участник может отправить до 5 идей в фазе сбора
- Идеи проходят фильтр нецензурной лексики
- Организатор может модерировать идеи перед запуском голосования
- Участник не может голосовать за свои идеи
- Результаты ранжированы по количеству голосов
- При равном количестве голосов — сортировка по времени подачи (раньше = выше)

---

### 3.10 Контент-контроль

**Описание:** Автоматическая фильтрация нецензурной лексики через словарь + ручная модерация организатором.

**Автоматическая фильтрация:**

- Словарь запрещённых слов хранится в файле `badwords.json` (русский + английский)
- Проверка применяется к нормализованному тексту (lowercase, без пробелов внутри слова)
- Совпадения заменяются на `***`
- Словарь расширяемый — администратор может добавлять слова

**Ручная модерация:**

- Ответы отображаются участникам сразу (не ждут одобрения)
- Организатор видит все открытые ответы в реальном времени
- Организатор может скрыть/удалить конкретный ответ
- Скрытый ответ исчезает из облака слов / списка идей, но сохраняется в БД с пометкой `hidden`
- Удалённый ответ не включается в экспортируемый отчёт

**Acceptance criteria:**

- Нецензурные слова из словаря автоматически маскируются
- Организатор может скрыть любой ответ в 1 клик
- Скрытые ответы не отображаются участникам, но доступны организатору
- Фильтр не блокирует отправку — только маскирует при отображении

---

### 3.11 Статистика в реальном времени (для организатора)

**Описание:** Панель организатора показывает статистику по текущему вопросу и сессии в целом.

**Отображаемые данные:**

- Количество подключённых / отключённых участников
- Количество ответивших / не ответивших на текущий вопрос
- Распределение ответов (bar chart для choice-вопросов)
- Среднее время ответа на текущий вопрос
- Текущий лидерборд

**Acceptance criteria:**

- Данные обновляются в реальном времени без перезагрузки
- Bar chart показывает распределение по вариантам ответа
- Процент ответивших отображается с прогресс-баром
- Организатор видит состояние каждого участника (ответил / не ответил / отключён)

---

### 3.12 Обратная связь участнику

**Описание:** Участник видит подтверждение приёма ответа, правильность (для квизов) и свою позицию в рейтинге.

**Acceptance criteria:**

- После отправки ответа отображается «Ответ принят»
- Для квизов: после завершения вопроса показывается правильный/неправильный ответ
- Для `speed_bonus`: показывается количество начисленных баллов
- После каждого вопроса участник видит свою позицию в рейтинге
- В конце сессии отображается итоговый результат и место

---

### 3.13 История сессий и личный кабинет

**Описание:** Организатор и участник могут просматривать историю проведённых/пройденных опросов.

**Для организатора:**

- Список проведённых сессий с датой, количеством участников, средним баллом
- Детальный просмотр результатов каждой сессии
- Возможность повторного экспорта результатов

**Для участника:**

- Список пройденных опросов (привязка по session_id, хранится в localStorage браузера)
- Результат и позиция в каждом пройденном опросе

**Acceptance criteria:**

- Организатор видит список всех проведённых сессий с основной статистикой
- Организатор может открыть детали любой прошедшей сессии
- Участник видит историю пройденных опросов (при сохранении session_id в браузере)

---

### 3.14 Экспорт результатов

**Описание:** Экспорт результатов сессии в CSV и PDF.

**CSV-экспорт:**

- Формат: UTF-8 с BOM для корректного открытия в Excel
- Колонки: participant_name, question_text, answer, is_correct, score, response_time_ms
- Эндпоинт: `GET /api/sessions/:id/export/csv`

**PDF-экспорт (красивый отчёт):**

- Титульная страница: название опроса, дата, количество участников
- Общая статистика: среднее время ответа, средний балл, процент правильных ответов
- По каждому вопросу: текст, распределение ответов (bar chart), правильный ответ
- Облако слов (если есть word_cloud вопросы) — как изображение
- Итоговый лидерборд (топ-10)
- Брендирование: логотип в хедере, цветовая схема
- Библиотека бэкенда: `jung-kurt/gofpdf` или `signintech/gopdf`
- Графики: рендеринг на фронтенде через `recharts`, экспорт в PNG через `html2canvas`, передача на бэкенд для вставки в PDF

**Acceptance criteria:**

- CSV содержит все ответы всех участников с корректной кодировкой
- PDF содержит графики, облако слов и лидерборд
- PDF визуально оформлен (не plain text)
- Экспорт доступен из страницы детализации сессии
- Файлы скачиваются через браузер

### 3.15 Презентации PowerPoint и синхронная демонстрация слайдов

**Описание:** Организатор загружает файл `.pptx` к своему опросу, сервер асинхронно конвертирует презентацию в набор изображений (по одному на слайд), а во время живой сессии организатор может открыть презентацию и перелистывать слайды — все подключённые участники видят тот же слайд в реальном времени. Функция заменяет ручное переключение между слайдами в PowerPoint и вкладкой браузера: ведущий остаётся в одном интерфейсе, а участники видят синхронизированный показ на своих устройствах.

**Поток загрузки:**

1. Организатор открывает свой опрос и в разделе «Презентации» загружает файл `.pptx` (max 50 МБ)
2. Фронтенд отправляет файл как `multipart/form-data` на `POST /api/presentations`
3. Бэкенд проверяет расширение `.pptx` и магические байты ZIP-заголовка (`50 4B 03 04`), отклоняя переименованные не-pptx файлы
4. Исходный файл сохраняется в приватный S3-бакет (MinIO), в БД создаётся запись презентации со статусом `processing`
5. Ответ `202 Accepted` возвращается мгновенно — дальнейшая конвертация идёт в фоновой горутине
6. Фронтенд опрашивает `GET /api/presentations/:id` и показывает прогресс до перехода в статус `ready` или `failed`

**Конвейер конвертации (фоновый worker):**

1. **LibreOffice headless** — `soffice --headless --convert-to pdf`: `.pptx` → `.pdf`
2. **Poppler `pdftoppm`** — `pdftoppm -r 150`: `.pdf` → `slide-1.png`, `slide-2.png`, …
3. **`cwebp`** — `cwebp -q 85`: каждый PNG → WebP (в среднем в 3–4 раза меньше PNG)
4. Каждый WebP загружается в публичный S3-бакет с ключом `presentations/{id}/slide-{n}.webp`
5. В БД создаются записи `presentation_slides` с позицией и S3-ключом, презентация переводится в статус `ready`
6. В случае ошибки на любом шаге статус становится `failed`, сохраняется `error_message` для отображения организатору

**REST API:**

- `POST /api/presentations` — загрузка `.pptx` (multipart/form-data, поля `file`, `title`, опционально `poll_id`), ответ `202` с объектом презентации
- `GET /api/presentations` — список презентаций организатора
- `GET /api/presentations/:id` — детали презентации + упорядоченный список URL слайдов (пусто, пока статус `processing`)
- `DELETE /api/presentations/:id` — удаление записи и каскадное удаление слайдов из S3

**WebSocket-протокол (организатор → сервер):**

```json
{"type": "open_presentation", "data": {"presentation_id": "uuid"}}
{"type": "change_slide", "data": {"position": 3}}
{"type": "close_presentation", "data": {}}
```

**WebSocket-протокол (сервер → все клиенты комнаты):**

```json
{"type": "open_presentation", "data": {"presentation_id": "uuid", "slides": ["url1", "url2", ...], "current_position": 0}}
{"type": "change_slide", "data": {"position": 3}}
{"type": "close_presentation", "data": {}}
```

**Снимок состояния при подключении:** если новый участник подключается к комнате, где уже открыта презентация, сервер сразу шлёт ему `open_presentation` с текущей позицией — участник не ждёт следующего перелистывания.

**UI организатора:**

- Плавающая кнопка «Открыть презентацию» (правый нижний угол) доступна во всех активных состояниях сессии (`waiting`, `active`, `showing_question`, `showing_results`) и скрыта в `finished`
- Клик раскрывает панель выбора из списка `ready`-презентаций организатора
- После открытия поверх экрана хоста появляется оверлей с крупным предпросмотром текущего слайда, стрелками перелистывания и счётчиком «3 из 12»; стрелки отправляют `change_slide` на сервер

**UI участника:**

- Когда организатор открывает презентацию, у участника появляется полноэкранный оверлей с изображением текущего слайда
- Оверлей НЕ блокирует приём ответов: если в этот момент идёт вопрос, участник может свернуть оверлей и ответить, а затем вернуться к просмотру
- При закрытии презентации оверлей исчезает у всех участников одновременно

**Acceptance criteria:**

- Организатор загружает `.pptx` до 50 МБ, видит статус обработки без блокировки UI
- Некорректные файлы (не `.pptx`, подделанное расширение, битый ZIP) отклоняются с понятной ошибкой
- После готовности слайд появляется у всех участников с задержкой < 500 мс после перелистывания организатором
- Новый участник, подключившийся к сессии с открытой презентацией, сразу видит актуальный слайд
- При ошибке конвертации презентация переводится в `failed` и показывает `error_message` организатору
- Удаление презентации каскадно удаляет все слайды из S3 и записи из БД

**Технические соображения:**

- Хранилище: MinIO (S3-совместимое) — публичный бакет для слайдов (anonymous read), приватный бакет для исходников `.pptx` (доступ только через backend с sigv4)
- В продакшене слайды отдаются через `nginx` по пути `/media/<bucket>/<key>` с `Cache-Control: public, max-age=2592000, immutable` и `proxy_cache` на 30 дней — участники грузят слайды через кэширующий прокси, минуя backend и не раскрывая MinIO наружу
- Зависимости образа backend: `libreoffice-core`, `poppler-utils`, `libwebp-tools` (устанавливаются в `backend/Dockerfile`)
- `S3_PUBLIC_BASE_URL` задаётся через переменную окружения: в dev — прямой `http://localhost:9000/<bucket>` (MinIO опубликован на хосте), в prod — относительный `/media/<bucket>` (резолвится браузером к тому же домену, что и фронтенд)
- Загрузка `.pptx` ограничена на двух уровнях: `client_max_body_size 150m` в nginx (многопартовый overhead) и `maxPresentationUploadSize = 50 << 20` в backend (фактический лимит)

## 6. Принципы UI-дизайна

### 6.1 Общие принципы

- **Mobile-first для участника:** интерфейс участника оптимизирован под экраны 320–428px
- **Desktop-first для организатора:** панель управления оптимизирована под экраны 1024px+
- **Минимум шагов:** участник подключается за 2 шага (ввод кода → ввод имени)
- **Крупные элементы на мобильном:** кнопки ответов не менее 48px, шрифт вопроса не менее 18px
- **Яркие цвета для вариантов ответа:** 4 контрастных цвета (как в Kahoot) для мгновенного визуального различия
- **Анимации:** плавные переходы между вопросами, анимация таймера, анимация лидерборда

### 6.2 Ключевые экраны

**Организатор:**

1. Дашборд — список опросов, кнопка «Создать»
2. Редактор опроса — конструктор вопросов с drag-and-drop
3. Экран ожидания — код комнаты, QR, список участников
4. Экран проведения — вопрос + статистика ответов в реальном времени
5. Экран результатов — лидерборд, детализация
6. История сессий — список прошедших сессий

**Участник:**

1. Вход — поле кода + поле имени
2. Ожидание — «Ждём начала...»
3. Вопрос — текст, варианты/поле ввода, таймер
4. Ожидание результата — «Ответ принят!»
5. Результат вопроса — правильно/неправильно, баллы, позиция
6. Итоги — финальный результат и место

### 6.3 Адаптивность

- Breakpoints: 320px (mobile), 768px (tablet), 1024px (desktop)
- Экран организатора при проведении должен хорошо смотреться при проецировании (1920×1080)
- Контрастность текста: WCAG AA (минимум 4.5:1)

## 7. Соображения по безопасности

### 7.1 Аутентификация

- Пароли: bcrypt с cost factor ≥ 10
- JWT access token: 15 минут, в Authorization header
- JWT refresh token: 7 дней, httpOnly secure cookie
- Rate limiting: 5 попыток логина/мин на IP

### 7.2 Валидация данных

- Все входные данные валидируются на бэкенде (не доверять клиенту)
- SQL injection: параметризованные запросы через sqlx
- XSS: экранирование пользовательского ввода при отображении
- CORS: разрешить только домен фронтенда

### 7.3 WebSocket

- Аутентификация при handshake (JWT для организатора, session_token для участника)
- Rate limiting на сообщения: не более 10 сообщений/сек от одного клиента
- Максимальный размер сообщения: 4 КБ
- Ping/Pong для обнаружения разрывов соединения (интервал 30 сек)

### 7.4 Загрузка файлов

**Изображения к вопросам:**

- Проверка MIME-type на сервере (не доверять Content-Type от клиента)
- Ограничение размера: 5 МБ
- Допустимые форматы: JPEG, PNG, WebP
- Генерация уникальных имён файлов (UUID)
- Хранение вне корневой директории веб-сервера

**Презентации `.pptx`:**

- Проверка расширения (`.pptx`) И магических байт ZIP-заголовка (`50 4B 03 04`) на сервере — переименованный `.exe` не пройдёт
- Ограничение размера: 50 МБ в backend (`MaxBytesReader`) + `client_max_body_size 150m` в nginx для multipart overhead
- Исходник сохраняется в приватный S3-бакет и не отдаётся напрямую участникам — только сконвертированные WebP-слайды попадают в публичный бакет
- Путь `/media/` в nginx `proxy_pass`-ит только на MinIO и отфильтровывает `x-amz-*` заголовки, чтобы не раскрывать метаданные хранилища

### 7.5 Контент-контроль

- Фильтрация нецензурной лексики на стороне бэкенда
- Модерация доступна только организатору конкретной комнаты

## 9. Потенциальные проблемы и решения

| Проблема                                                        | Решение                                                                                                                                                      |
| ----------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Синхронизация таймера между клиентами | Серверный таймер с broadcast через WebSocket, клиент корректирует drift по серверным тикам                   |
| Потеря WebSocket-соединения                             | Exponential backoff reconnect на клиенте, восстановление состояния по session_token                                               |
| Конкурентная запись ответов                    | Unique constraint (participant_id, question_id) + проверка временного окна на сервере                                                |
| Масштаб: много комнат одновременно        | Каждая комната — изолированный Hub в горутине, Redis pub/sub при горизонтальном масштабировании |
| Медленная генерация PDF                               | Генерация в фоновой горутине, уведомление организатору по готовности                                    |
| Фильтрация обсценной лексики                  | Словарь + нормализация; обновляемый словарь без рестарта сервера                                             |
| Одинаковые имена участников                    | Различение по внутреннему session_id, имя — только для отображения                                                   |
| Большое облако слов                                    | Ограничение топ-100 слов для визуализации, остальные — в агрегированных данных                      |

## Приложение В: CI/CD Pipeline (GitHub Actions)

```yaml
name: Deploy to VPS

on:
  push:
    branches: [master]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build and push Docker images
        run: |
          docker compose -f docker-compose.prod.yml build

      - name: Deploy to VPS
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.VPS_HOST }}
          username: ${{ secrets.VPS_USER }}
          key: ${{ secrets.VPS_SSH_KEY }}
          script: |
            cd /opt/pollapp
            git pull origin master
            docker compose -f docker-compose.prod.yml up -d --build
            docker system prune -f
```
