# Нагрузочное тестирование Presentarium

Скрипт `room_load_test.js` реализует сценарий нагрузочного тестирования
из таблицы 3.6: одна комната, один организатор, пять вопросов
(`single_choice` + `open_text` + `word_cloud`), 30 секунд активного времени
на вопрос. Скрипт сам поднимает данные сценария (регистрирует организатора,
создаёт опрос, вопросы и комнату), запускает `N` участников по WebSocket,
собирает метрики и в конце печатает таблицу «критерий → факт → PASS/FAIL»,
напрямую соответствующую таблице 3.7.

## Установка k6

- macOS: `brew install k6`
- Linux: см. https://grafana.com/docs/k6/latest/set-up/install-k6/
- Windows: `winget install k6` или `choco install k6`

Минимальная версия — k6 0.43.

## Подготовка backend'а

Для боевого запуска нужно поднять стек так же, как в production: PostgreSQL,
backend на порту 8080, миграции применены. Самый простой путь:

```bash
docker compose up -d
```

либо:

```bash
cd backend && go run ./cmd/server
```

База должна быть пустой или допускать создание новых пользователей —
скрипт регистрирует организатора с уникальным email (`loadtest-<rand>@k6.test`),
так что повторные запуски не конфликтуют.

## Запуск

Из директории `loadtest/`:

```bash
# Уровни нагрузки из таблицы 3.6
k6 run -e PARTICIPANTS=50  room_load_test.js
k6 run -e PARTICIPANTS=100 room_load_test.js
k6 run -e PARTICIPANTS=250 room_load_test.js
k6 run -e PARTICIPANTS=500 room_load_test.js
```

Полный набор переменных окружения:

| Переменная       | По умолчанию              | Назначение                                                       |
| ---------------- | ------------------------- | ---------------------------------------------------------------- |
| `BASE_URL`       | `http://localhost:8080`   | HTTP/WS endpoint backend'а                                       |
| `PARTICIPANTS`   | `50`                      | Сколько участников запускать одновременно                        |
| `QUESTION_TIME`  | `30`                      | Длительность активного вопроса (сек). Сервер автоматически закроет вопрос по таймеру |
| `STARTUP_BUFFER` | `max(5, PARTICIPANTS/50)` | Сколько секунд организатор ждёт, пока участники приконнектятся, прежде чем запустить первый вопрос |

При высоких уровнях нагрузки на машине-генераторе понадобится поднять лимит
файловых дескрипторов (Linux/macOS):

```bash
ulimit -n 4096
```

На Windows ограничение по эфемерным портам по умолчанию ~16k — для 500
участников этого хватает.

## Что измеряется и как это маппится на таблицу 3.7

| №  | Критерий                                                                | Как проверяется                                                                                                                                  | Метрика k6                |
| -- | ----------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------- |
| 1  | До 500 участников в комнате                                             | Конфигурацией запуска (`PARTICIPANTS=500`)                                                                                                       | —                         |
| 2  | ≥99 % успешных подключений                                              | Каждый участник, получивший событие `connected`, увеличивает rate; обрыв до handshake считается провалом                                          | `connected_rate`          |
| 3  | Средняя задержка доставки вопроса ≤500 мс                               | `Date.now() - data.sent_at_ms`, где `sent_at_ms` — серверный timestamp в самом броадкасте `question_start`                                       | `wait_for_question_ms`    |
| 4  | Среднее время подтверждения ответа ≤500 мс                              | Время от отправки `submit_answer` / `submit_text` до получения `answer_accepted`                                                                  | `answer_ack_ms`           |
| 5  | Восстановление после переподключения ≤3 с                               | Покрыто интеграционным тестом `TestParticipantReconnect` в `backend/test/integration/coverage_gaps_test.go` (там же замер логирован) — k6 этот сценарий не проверяет | (вне k6)                  |
| 6  | Доля ошибок при отправке ответов ≤1 %                                   | Любая исключение при `socket.send` помечает submit как ошибку                                                                                    | `submit_error_rate`       |
| 7  | 0 потерянных ответов                                                    | Если `submit_*` отправлен, но `answer_accepted` не пришёл до `question_end` — счётчик инкрементируется                                            | `lost_answers`            |
| 8  | 0 повторно учтённых ответов                                             | Если получено два `answer_accepted` для одного `question_id` у одного участника — счётчик инкрементируется                                       | `duplicate_acks`          |
| 9  | Аварийное завершение серверного приложения не допускается               | Out-of-band: следить за логом backend'а во время прогона, проверить, что процесс жив после теста                                                  | (вне k6)                  |
| 10 | Некорректное завершение WS-соединений без действия пользователя         | Out-of-band, плюс `session_end_received` rate как индикатор: участник должен либо получить `session_end` от организатора, либо это считается провалом | `session_end_received`    |
| 11 | Сохранение результатов после завершения сессии                          | В `teardown()` вызывается `GET /api/sessions` — проверяется, что наш sessionId есть в списке                                                      | check `session in history`|
| 12 | Формирование итогового лидерборда                                       | В `teardown()` вызывается `GET /api/sessions/{id}` — проверяется, что `leaderboard` не пуст и отсортирован по `total_score` по убыванию           | check `leaderboard ...`   |

Все «жёсткие» критерии (2, 3, 4, 6, 7, 8) подняты как k6 thresholds — при
их нарушении `k6 run` завершится с ненулевым exit code, что удобно для CI.

## Что выдаст скрипт

После прогона в stdout появится:

1. Стандартный вывод k6 (метрики `http_req_*`, `ws_*`, `data_*`).
2. Раздел `Presentarium load test — criterion summary (Table 3.7)` с таблицей:
   ```
   Presentarium load test — criterion summary (Table 3.7)
   ──────────────────────────────────────────────────────
   №2  Successful connections (≥99%)        99.80%   PASS
   №3  Question delivery avg (≤500 ms)      143.21 ms   PASS
   №4  Answer ack avg (≤500 ms)             227.04 ms   PASS
   №6  Submit error rate (≤1%)              0.20%    PASS
   №7  Lost answers (=0)                    0        PASS
   №8  Duplicate answer_accepted (=0)       0        PASS
        session_end received                100.00%  PASS
   ──────────────────────────────────────────────────────
   ```
3. Файл `summary.json` рядом со скриптом — полный JSON всех метрик k6,
   удобно для последующего разбора в Python / Grafana и т.п.

## Известные ограничения

- **Reconnect (критерий 5)** в k6-сценарии не моделируется — это отдельный
  поведенческий тест; он есть в `backend/test/integration/coverage_gaps_test.go`
  (`TestParticipantReconnect`).
- **Здоровье сервера (9, 10)** скрипт проверить не может — нужен внешний
  мониторинг (`docker logs -f backend`, `systemctl status`, etc.). Если
  процесс упадёт, прогон k6 это увидит косвенно через лавину обрывов
  (`connected_rate` / `session_end_received` уйдут в пол).
- При запуске на той же машине, что и backend, и при `PARTICIPANTS=500`
  k6 + сервер борются за CPU — для чистой картины разносите генератор
  и backend на разные хосты.

## Stress-тест: ramp до отказа + графики

Скрипт `stress_test.js` решает другую задачу: вместо PASS/FAIL по
фиксированному N он плавно наращивает число участников от 0 до
`MAX_PARTICIPANTS` (по умолчанию 2000), чтобы найти точку, в которой
система деградирует или отказывает. **Thresholds в нём нет специально** —
прогон всегда завершится с exit code 0, чтобы k6 не оборвал сбор метрик
на самом интересном месте; «предел» определяется визуально по графикам.

Профиль нагрузки по умолчанию (~10 минут):

| Стадия     | Длительность | Целевое число VU         |
| ---------- | ------------ | ------------------------ |
| Ramp 1     | 60 s         | 5 % от MAX (≈100)        |
| Ramp 2     | 60 s         | 12.5 %                   |
| Ramp 3     | 60 s         | 25 %                     |
| Ramp 4     | 120 s        | 50 %                     |
| Ramp 5     | 120 s        | 75 %                     |
| Ramp 6     | 120 s        | 100 % (MAX)              |
| Hold peak  | 60 s         | MAX                      |
| Drain      | 30 s         | 0                        |

Организатор всё это время крутит 5 вопросов по кругу (по 15 секунд каждый
по умолчанию), так что у любого VU, подключившегося в любой момент,
всегда есть активный вопрос.

### Запуск с web-dashboard

k6 умеет рисовать графики в реальном времени и сохранять полный HTML-отчёт
(встроено с k6 v0.49). Из директории `loadtest/`:

```bash
# Linux / macOS
K6_WEB_DASHBOARD=true \
K6_WEB_DASHBOARD_EXPORT=report.html \
k6 run stress_test.js
```

```powershell
# Windows PowerShell
$env:K6_WEB_DASHBOARD="true"
$env:K6_WEB_DASHBOARD_EXPORT="report.html"
k6 run stress_test.js
```

Что произойдёт:

1. Пока тест идёт — на `http://127.0.0.1:5665` доступна live-панель с
   графиками (число VU, latency, error rate во времени).
2. После завершения рядом со скриптом появится **`report.html`** — один
   самодостаточный HTML с интерактивными графиками. Его удобно открывать
   в браузере и делать скриншоты для дипломной работы.
3. Дополнительно сохраняется `stress_summary.json` — полный JSON-снимок
   итоговых метрик (peak VUs, p95, error rate и т.д.).

### Альтернативные параметры запуска

```bash
# Свой потолок (например, 1500)
k6 run -e MAX_PARTICIPANTS=1500 stress_test.js

# Более короткие вопросы → больше циклов на тех же 10 минутах
k6 run -e QUESTION_TIME=10 stress_test.js

# Дым-тест профиля (около минуты, для проверки самого скрипта)
k6 run -e STAGE_PROFILE=smoke stress_test.js
```

### Что искать на графиках

| Метрика на дашборде          | Что показывает                                                    |
| ---------------------------- | ----------------------------------------------------------------- |
| `vus`                        | Текущее число активных участников — ось «нагрузка» во времени      |
| `wait_for_question_ms` p95   | Задержка доставки вопроса. Пока линейна — система держит. Где взлёт — практический предел |
| `answer_ack_ms` p95          | Время round-trip ответа. Похожая интерпретация                     |
| `connected_rate`             | Доля успешных подключений. Падение ниже 0.99 ≈ отказ обслуживания  |
| `submit_error_rate`          | Доля ошибок при `socket.send`. Скачок = WS-канал перегружен        |
| `lost_answers` / `ws_errors` | Кумулятивные счётчики. Резкий рост = серверу плохо                 |
| `ws_session_duration` (k6)   | Если короче ожидаемого — соединения принудительно закрываются      |

Типичная «подпись» отказа: при ~N VU `vus` всё ещё растёт по плану, но
`wait_for_question_ms` p95 уходит из ~150 мс в сотни и тысячи, `connected_rate`
проседает ниже 0.99, `lost_answers` начинает накапливаться. Это и есть
заявляемый предел вместимости одной комнаты.

### Лимиты ОС для 2000 VU

- Linux / macOS: `ulimit -n 8192` перед запуском (по WS на VU открывает 1
  сокет, плюс HTTP, плюс служебные fd — 4096 уже впритык).
- Windows: эфемерные порты по умолчанию ~16k — хватает; при необходимости
  поднять диапазон через `netsh int ipv4 set dynamicport`.
- 2000 WebSocket-сессий на одном генераторе требуют ≥2 ГБ RSS у k6 — при
  меньших объёмах памяти лучше снизить `MAX_PARTICIPANTS` или разнести
  k6 и backend на разные хосты.
